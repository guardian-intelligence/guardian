# Secrets: the model in five minutes

Orientation doc for configuring secrets per environment. Deep references:
`docs/openbao-design.md` (full design + trade-offs) and
`src/infrastructure/runbooks/openbao-static-seal-self-init.md` (exact commands,
bootstrap, DR, re-initialization).

## Bootstrap

- A 32-byte static seal key, generated by a repo-pinned tool into offline
  custody, is placed out of band on the three dedicated, tainted, key-bearing
  nodes. It never appears in Git, Kubernetes, CI, chat, shell history, or
  OpenBao itself.
- cert-manager (an independent listener CA, not OpenBao PKI) issues the API
  listener cert before OpenBao starts; OpenBao self-unseals from the key file
  on every start. "Sealed" is a fault, not a resting state.
- OpenBao **self-init** runs its `initialize` block exactly once at first
  cluster initialization and is the **sole source of truth** for OpenBao
  config: Kubernetes auth, the `kv`/`transit` mounts, and every policy/role.
  The temporary privileged token is revoked by OpenBao afterward. No root
  token or recovery keys exist; break-glass is regenerate-from-custody.
- The bootstrap importer (`src/infrastructure/cmd/openbao_secret_import`)
  moves custody values into KV at bootstrap or DR re-seed through a
  self-deleting wildcard role. Its import plan in `main.go` **is the DR
  re-seed manifest** — the authoritative list of every secret the system
  needs after a raft wipe.

## Runtime topology

- **One OpenBao instance for the whole cluster** — a 3-node raft StatefulSet
  in `tenant-guardian` on node-local storage (raft replicates at the
  application layer). Stage isolation is at the **policy layer, not the
  instance layer**: beta/gamma/prod are namespaces in one physical cluster,
  so per-stage instances would share the same nodes anyway.
- Workloads never talk to OpenBao. External Secrets Operator materializes
  native Kubernetes Secrets (`creationPolicy: Orphan` /
  `deletionPolicy: Retain`, 1h refresh), so an OpenBao outage degrades secret
  *rotation*, not running consumers.

## Permission model

- Access is scoped **per consumer namespace subtree**:
  `kv/guardian/guardian-mgmt/<namespace>/*`. Every namespace gets exactly one
  pair:
  - `guardian-reader-<ns>` — SA `secrets-reader`, read-only, what ESO
    authenticates as (1h token TTL);
  - `guardian-writer-<ns>` — SA `secrets-writer`, write-only within the same
    subtree, 10-minute token TTL, mintable only with custody kubeconfig
    access. The SAs mount nothing and run nothing.
- OpenBao config is therefore **O(1) in the number of integrations**: a new
  secret in an existing namespace never touches OpenBao config. Only
  *structural* changes (a new consumer namespace, a new mount or auth method)
  edit the self-init block and re-initialize (raft reset + custody re-import;
  zero consumer downtime because materialized Secrets are Orphan/Retain).
- Steady state holds **zero standing admin**: no credential exists that can
  read or write across namespaces.

## Mistakes prevented by construction

- **Prod/gamma mixups are impossible server-side**: a writer token for one
  stage physically cannot write another stage's paths — OpenBao denies it,
  care is not involved.
- **Repo-side, the same invariant is CI-enforced**:
  `TestOpenBaoSecretScopeConformance` rejects any `ExternalSecret` remoteRef
  or vault `SecretStore` outside the consuming namespace's own subtree, so a
  beta manifest referencing a prod path cannot merge.
- **No plaintext in Git** — manifests carry only remoteRefs and policy names.
  `${...}` placeholders that must survive Flux envsubst are guarded by
  `TestFluxSubstitutionSafetyConformance` (unannotated ones get silently
  blanked — the test catches this at review time).
- **No standing admin to leak**: the self-init token is revoked, the importer
  deletes its own role and policy, and recovery keys were never created.
- **Values never touch argv or shell history**: writes go through
  `bao kv put ... key=-` (stdin) or `@file.json` under `umask 077`.
- **DR cannot forget a secret**: the importer plan is extended in the same PR
  that adds the integration, so the re-seed manifest stays authoritative by
  review, not by memory.

## Open risks (stated, not hidden)

- The static seal key is a node-local bearer secret: **root compromise of a
  key-bearing node is OpenBao compromise**. Accepted posture (unattended
  restart, simple DR); compensated by tainted dedicated nodes and hostPath
  admission control, not eliminated.
- The single instance means an **OpenBao version upgrade hits all stages at
  once** — there is no staged rehearsal like Keycloak's promotion chain.
  Mitigations: ESO caching decouples consumers; rehearse upgrades by
  restoring the latest raft snapshot into a throwaway instance first.
- **Rotation is manual today.** The writer path makes rotation cheap and
  safe, but nothing rotates automatically or alerts on secret age/expiry yet
  ("autoprovisioned/autorotated" is the constraint; rotation automation is
  the open half).
- **Snapshot/restore automation is deferred** (pending Velero research);
  manual `bao operator raft snapshot` is the known primitive. Snapshots are
  barrier-encrypted — worthless without the seal key, so key custody couples
  to snapshot retention.
- The custody env file (`custody.env`) is the DR re-seed root. It lives
  inside the encrypted custody bundle (`aspect infra custody`,
  `runbooks/custody.md`); plaintext exists only at
  `/dev/shm/guardian-custody/custody.env` between an explicit restore and
  wipe, never at rest in the operator home.

## OpenBao vs a plain Kubernetes Secret

The dividing line is **where the value's source of truth lives** — ask
"after a cluster wipe, where does this come back from?":

- **OpenBao (via ExternalSecret)** — the source of truth is *outside* the
  cluster and the cluster cannot regenerate it: third-party API keys, OAuth
  client secrets, GitHub App private keys, anything a human or agent obtained
  from an external system. These must survive DR, so they live in custody,
  re-seed through the importer plan, and distribute through OpenBao's scoped,
  audited, server-side-enforced write path.
- **Operator-generated Kubernetes Secret** — the credential is *born inside*
  the cluster and both ends live here: CNPG role passwords
  (e.g. `postgres-keycloak-app`), ClickHouse users, inter-service tokens
  minted by an operator. DR for these is **regenerate, never restore**: no
  custody entry, no importer-plan line, no ExternalSecret. Routing them
  through OpenBao would add a custody loop and a dependency for a value
  nobody outside the cluster ever needs — ESO's output is a Kubernetes
  Secret anyway, so nothing is gained at rest.
- **No credential at all** — some services authenticate by network position,
  not secrets (TigerBeetle has no authn layer; access control is the Cilium
  policy). Don't invent a credential to decorate a network boundary.

Corollaries:

- Never add an in-cluster-generated credential to the custody env file or
  the importer plan; if DR can regenerate it, a custody listing is a bug (a
  stale copy that will one day be wrong).
- Rotation cadence is bounded by the consumer's pickup path: mounted Secret
  files update in place, **env-var consumers need a pod roll**. Prefer
  eliminating the credential (CNPG TLS client-cert auth, Kubernetes auth)
  over rotating it fast.
- The scope conformance test governs everything OpenBao-backed; the
  in-cluster class is convention — reviewers should reject PRs that project
  operator-generated credentials through OpenBao.

## Adding a secret for a third-party integration (routine)

No OpenBao changes. One PR plus one scoped write — full commands in the
runbook's "Adding An Integration" section:

- **Git PR**: add the `ExternalSecret` (+ workload wiring) in the consuming
  namespace reading `guardian/guardian-mgmt/<namespace>/<integration>`, and
  extend the importer plan in `openbao_secret_import/main.go` (+ its test) so
  DR re-seeds the new value.
- **Custody**: `aspect infra custody --action restore`, append the value to
  `/dev/shm/guardian-custody/custody.env` (never echo it), then
  `--action create` to snapshot and `--action wipe`. Refresh the offline
  copies — this is a custody event.
- **Value write**: mint a short-lived `secrets-writer` token for that
  namespace and `bao kv put` with the value on stdin. The role cannot write
  outside its subtree, so the wrong-stage failure mode does not exist.
- **Verify**: the `ExternalSecret` reports `Ready=True` (force-sync
  annotation to refresh immediately).
